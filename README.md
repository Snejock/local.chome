# chome — домашний стек на Docker Compose

Этот репозиторий содержит набор самостоятельных сервисов, запускаемых через Docker Compose в единой пользовательской сети. Цель — развернуть домашнюю инфраструктуру (медиа, хранилище, обратный прокси, DNS и т. п.) единообразно и воспроизводимо.

> Важно: README описывает общие подходы и процессы. Состав конкретных контейнеров и их настройки могут со временем меняться. Не привязывайтесь к перечню сервисов — ориентируйтесь на файлы в каталоге `compose/` и корневой `docker-compose.yml`.

## Требования
- Linux/WSL2/macOS с установленными:
  - Docker Engine 24+
  - Docker Compose V2 (рекомендуется ≥ 2.20 для поддержки `include`)
- Доступ к терминалу и базовые знания Docker
- Достаточно свободного места на диске под тома данных

## Быстрый старт
1. Клонируйте репозиторий:
   - `git clone <url> && cd chome`
2. Проверьте и при необходимости отредактируйте переменные в `.env` (например, доступ к БД и пароли).
3. При необходимости включите/отключите нужные сервисы в корневом `docker-compose.yml` (строки `include:` можно комментировать/раскомментировать).
4. Запустите стек:
   - `docker compose up -d`
5. Проверьте состояние:
   - `docker compose ps`
   - `docker compose logs -f` (для просмотра логов всего стека) или `docker compose logs -f <service>`

## Как это устроено
- В корне лежит `docker-compose.yml`, который с помощью директивы `include` агрегирует файлы из каталога `compose/<service>/docker-compose.yml`.
- Все сервисы подключаются к общей сети `chome-net` (bridge), что упрощает их взаимодействие.
- Переменные окружения берутся из `.env` в корне репозитория (например, учетные данные для БД).
- Данные сервисов, как правило, вынесены в постоянные тома/директории на хосте (см. соответствующие compose-файлы). По умолчанию путь может выглядеть как `/mnt/data/projects/chome/volumes/...` — проверьте и скорректируйте под свои нужды.

## Обычные операции
- Запуск/обновление конфигурации:
  - `docker compose up -d`
- Остановка всего стека:
  - `docker compose stop`
- Полная остановка и удаление контейнеров (данные в томах сохраняются):
  - `docker compose down`
- Обновление образов и перезапуск:
  - `git pull`
  - `docker compose pull`
  - `docker compose up -d`
- Логи:
  - `docker compose logs --since=1h`
  - `docker compose logs -f <service>`

## Настройка и переопределения
- Переменные в `.env` переопределяют значения внутри compose-файлов. Не храните в нем секреты без необходимости; используйте менеджер секретов или защищайте файл.
- Если нужно изменить каталоги для данных (volumes/bind mounts), правьте соответствующие `volumes:` в `compose/<service>/docker-compose.yml`.
- Чтобы временно отключить сервис, закомментируйте его `include` в корневом `docker-compose.yml` и выполните `docker compose up -d`.

## Резервное копирование
- Резервируйте каталоги томов, указанные в `volumes:` каждого сервиса. Примерно это пути вида `/mnt/data/projects/chome/volumes/...`.
- Для горячего бэкапа БД используйте экспорт/дамп (например, `pg_dump`), либо останавливайте соответствующий сервис перед копированием данных.

## Отладка и типичные проблемы
- Конфликты портов: измените публикации портов в нужном compose-файле и перезапустите стек.
- Переменные не подставляются: убедитесь, что запускаете из корня репозитория и файл `.env` существует, без лишних пробелов и с корректной кодировкой.
- Сервис «здоров», но недоступен: проверьте брандмауэр/маршрутизацию на хосте, а также правильность адресов и портов.
- Ошибка при `include`: удостоверьтесь, что используете Docker Compose V2 с поддержкой `include` (например, `docker compose version`).

## Безопасность
- Не публикуйте `.env` и другие секреты в публичные репозитории.
- Ограничивайте доступ к хосту, на котором разворачивается стек.
- По возможности используйте уникальные пароли и регулярные обновления образов.
